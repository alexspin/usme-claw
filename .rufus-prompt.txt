BACKGROUND

The USME reflection pipeline (reflect.ts) runs twice daily and proposes new skill
candidates into the skill_candidates table. The problem: it never looks at what's
already pending — it only checks the skills table (which is currently empty). After
158 candidates have accumulated, the table is full of near-duplicates like
"Fix PostgreSQL Transaction Cascade Errors", "Handle PostgreSQL Savepoint Poisoning",
"Debug PostgreSQL Nested Transaction Failures" — all the same pattern, proposed again
on every reflection run.

Three changes are needed. They must be implemented cleanly with tests written before
the implementation is considered done.

---

REQUIREMENTS

**Change 1 — Migration 016: Enable pg_trgm and add similarity index**

pg_trgm is a standard Postgres extension that scores string similarity by comparing
trigrams. It ships with every standard Postgres install.

Create file: packages/usme-core/db/migrations/016_pg_trgm.sql

Contents:
- CREATE EXTENSION IF NOT EXISTS pg_trgm;
- CREATE INDEX IF NOT EXISTS idx_skill_candidates_name_trgm
    ON skill_candidates USING gin (name gin_trgm_ops)
    WHERE dismissed_at IS NULL;

Migration must be idempotent (IF NOT EXISTS on both statements).
Do NOT add this migration to any auto-run list — user applies migrations manually.

**Change 2 — Corpus fetch: inject active candidates into prompt**

In runReflection() in packages/usme-core/src/consolidate/reflect.ts:

The corpus fetch phase currently queries:
  SELECT name FROM skills WHERE status = 'active'
to tell the LLM "these skills already exist, don't duplicate them".

Add a 6th query immediately after:
  SELECT id, name, description FROM skill_candidates
  WHERE dismissed_at IS NULL
  ORDER BY created_at DESC

Both lists must be injected into the LLM prompt with distinct labels:

  Active skills (already promoted — do NOT reproduce these):
  [list]

  Pending review queue (already in the candidate backlog — do NOT propose near-duplicates of these):
  [list with id, name, description]

If the candidates list is empty, inject a placeholder: "(none yet)" — do not crash or skip
the section.

**Change 3 — Output schema: add candidate_dismissals + response handler**

The reflection tool schema currently has four arrays:
  concept_updates, new_skills, contradictions, entity_updates

Add a fifth:
  candidate_dismissals: array of { candidate_id: number, reason: string }

Valid dismissal reasons (include these in the prompt as guidance):
  - "too specific: single incident, will not recur"
  - "duplicate: covered by [candidate name]"
  - "micro-pattern: not generalizable across domains"

In the response handler, process candidate_dismissals BEFORE new_skills:
  - For each entry, run:
      UPDATE skill_candidates
      SET dismissed_at = NOW(), updated_at = NOW()
      WHERE id = $1
  - Wrap in the same SAVEPOINT isolation pattern already used for concept_updates
    (SAVEPOINT before, RELEASE SAVEPOINT after success, ROLLBACK TO SAVEPOINT on error)
  - Log how many were dismissed: "Dismissed N candidates"
  - Include dismissalsProcessed count in the returned result object

**Change 4 — pg_trgm similarity guard at insert time**

Before each INSERT INTO skill_candidates for a proposed new skill, run a pre-check:
  SELECT id, name FROM skill_candidates
  WHERE dismissed_at IS NULL
  AND similarity(name, $1) > 0.5
  ORDER BY similarity(name, $1) DESC
  LIMIT 1

If a row is returned: skip the INSERT, log the skip:
  "Skipping '[proposed name]' — too similar to existing candidate '[existing name]' (trgm guard)"

If no row: proceed with INSERT as normal.

This is the last-resort guard for LLM misses. The ON CONFLICT (name) DO NOTHING
safety catch must stay in place too.
