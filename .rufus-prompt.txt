## BACKGROUND & CONTEXT

The USME (Utility-Shaped Memory Ecology) system is a multi-tier agent memory system
built on PostgreSQL + pgvector. It stores sensory traces, episodes, concepts, entities,
skills, and constraints. A nightly reflection process (reflect.ts) assembles the entire
memory corpus, sends it to Claude Sonnet via tool_use, and applies structured updates:
consolidating concepts, surfacing skill candidates, resolving contradictions, building
entity relationships, and extracting behavioral constraints.

### The problem with reflect.ts today

reflect.ts has grown to 1,020 lines and does five unrelated things in a single file:
1. Fetches the corpus from Postgres (concepts, episodes, traces, entities, skill candidates)
2. Builds slug-to-UUID index maps (so the LLM sees short readable labels instead of UUIDs)
3. Assembles the massive prompt string and calls Claude
4. Parses and validates the LLM response (Zod + jsonrepair recovery)
5. Writes all results back to Postgres in a transaction with SAVEPOINT isolation

This makes the file very hard to maintain, reason about, and test. Numbers like
"5 entity updates per run" or "350K token threshold" are buried inline. The prompt
text is inline. All config is scattered. The entity relationship cap (5/run) is far
too low now that the slug-remapping bug has been fixed — the entity graph has 1,311
entities but only 27 relationships, mostly because relationships were silently failing
due to the UUID bug and because the cap is too restrictive.

### The entity graph crisis

The entity extractor runs on every turn and has accumulated 1,311 entity rows, but the
relationship table has only ~27 valid edges. Most entities are orphans — they were
created but never connected. Worse, ~830 of those entities are pure noise: file paths
like "scripts/backfill-importance.mts", migration filenames like "018_entity_rel_unique.sql",
commit hashes like "1324110", and concept-type entities (which belong in the concepts
table, not entities). These noise rows will never generate meaningful relationships and
pollute every reflection run.

The reflect.ts entity_updates block processes at most 5 entities per run from the 48-hour
trace window only — which means it cannot systematically build the graph for the 1,200+
orphaned entities outside that window.

### The constraint duplication problem

The constraints table now has 10 rows but 5 are duplicates — the same behavioral rules
extracted twice across two reflection runs. This happens because the reflection prompt
has no visibility into existing active constraints, so the LLM re-extracts semantically
identical rules every time. The fix requires injecting the current active constraint list
into the prompt before the constraint extraction section, plus a pg_trgm similarity gate
at insert time.

### What we're building

1. **Refactor reflect.ts into focused modules** (pure behavior-preserving split):
   - reflect-config.ts — all tunable numbers
   - reflect-corpus.ts — DB fetch + slug index builder
   - reflect-prompts.ts — all prompt-building functions
   - reflect-writes.ts — all DB write logic
   - reflect.ts — thin orchestrator (~200 lines)

2. **Fix constraint deduplication** — inject active constraints into prompt + pg_trgm insert guard

3. **Raise entity relationship cap** from 5 to 20 per run

4. **Hard-delete noise entities** (file paths, migration names, concept-type orphans with no relationships)

5. **Build graph-builder.ts** — a new module that runs AFTER reflection as part of the same
   schedule. Unlike the entity_updates in reflect.ts (which only sees the 48h window and is
   capped at 5-20), the graph builder processes ALL entities in the DB, prioritizing orphans,
   batching them 50 at a time, asking the LLM to find directional relationships evidenced by
   the corpus, and writing them with the same SAVEPOINT pattern.

6. **Wire graph-builder into scheduler.ts** — call runGraphBuilder() after runReflection()
   completes on each scheduled run

7. **Write tests with mock data** — both the corpus/slug system and graph-builder need
   unit tests so we can verify behavior without hitting the live DB or LLM.

### graph-builder.ts (new file)

```typescript
export interface GraphBuilderOptions {
  model?: string;
  dryRun?: boolean;
  triggerSource: string;
}
export interface GraphBuildResult {
  batchesRun: number;
  entitiesProcessed: number;
  relationshipsWritten: number;
  durationMs: number;
}
export async function runGraphBuilder(opts: GraphBuilderOptions): Promise<GraphBuildResult>
```

Implementation:

1. Fetch ALL entities ordered by relationship count ascending (orphans first):
```sql
SELECT e.id, e.name, e.entity_type, e.canonical,
       COUNT(r.id) as rel_count
FROM entities e
LEFT JOIN entity_relationships r ON r.source_id = e.id AND r.valid_until IS NULL
WHERE e.exclude_from_reflection = false
GROUP BY e.id
ORDER BY rel_count ASC, e.created_at ASC
```

2. Fetch evidence corpus:
```sql
SELECT id, content, memory_type FROM sensory_trace
WHERE exclude_from_reflection = false
  AND created_at > NOW() - INTERVAL '48 hours'
ORDER BY created_at DESC LIMIT 500
```
```sql
SELECT id, summary FROM episodes
WHERE exclude_from_reflection = false
ORDER BY access_count DESC LIMIT 60
```

3. Format evidence corpus as tracesText / episodesText strings (same format as reflect-corpus.ts)

4. Batch entities into groups of GRAPH_BUILDER_CONFIG.batchSize (50)

5. For each batch:
   a. Build entity slug index for the batch (reuse makeSlug/assignSlug from reflect-corpus.ts)
   b. Format as: [entity:slug] Name (type) existing_rels=N
   c. Call Sonnet with buildGraphBuilderPrompt(batch, tracesText, episodesText, config)
      Tool schema: graph_output with relationships array:
        { source_slug: string, relationship: enum[allowed verbs], target_slug: string, confidence: number }
   d. For each proposed relationship:
      - Remap source_slug via batch entitySlugIndex
      - Remap target_slug via batch entitySlugIndex; if not found, try:
        SELECT id FROM entities WHERE lower(name) = lower($1) LIMIT 1
        as a name-based fallback before skipping
   e. Write with SAVEPOINT isolation (reuse writeEntityUpdates from reflect-writes.ts)
   f. Log per-batch: N/total batches, entities in batch, proposed, written

6. Return GraphBuildResult

buildGraphBuilderPrompt content (in reflect-prompts.ts):
```
You are building a relationship graph for an AI memory system called USME.

You are shown a batch of named entities and a corpus of recent memory evidence.
Your job: identify which relationships between these entities are clearly evidenced
by the corpus.

## Evidence Corpus

### Recent Traces (last 48h)
{tracesText}

### Recent Episodes
{episodesText}

## Entity Batch ({count} entities)
{entitiesText}

## Rules
- Only propose a relationship evidenced by at least 2 distinct traces or episodes
- Only use these verbs: uses, manages, owns, part_of, calls, routes_via, works_at, is_a
- Do NOT use: related_to, related, associated_with, or any other vague verb
- Use [entity:slug] labels exactly as shown — do not invent slugs
- target_slug can reference entities NOT in this batch if they appear in the evidence corpus
- Return [] if no relationships meet the evidence bar

Call graph_output with:
- relationships: array of { source_slug, relationship, target_slug, confidence }
  confidence = 0.5–1.0 based on evidence strength
```
